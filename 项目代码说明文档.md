# 报告系统看板代码说明文档

## 1. 项目概述

本项目是一个面向实验室业务场景的前端原型系统，整体名称在页面中显示为“智慧化实验室管理系统 · 原型”。它通过单页应用的方式，将实验室总览、测序质控、结果质控、人效分析、科研画图服务、实验室管理等多个业务模块集中到同一个界面中展示。

当前代码实现的核心定位是：

- 一个纯前端原型，不依赖后端接口
- 通过模拟数据演示业务看板、筛选、统计和图表交互
- 适合用于方案汇报、原型演示、需求讨论和后续产品化设计

## 2. 技术实现方式

项目没有使用 React、Vue 等框架，而是采用了最直接的原生前端实现：

- `index.html`
  负责承载所有页面结构、模块容器、表格、按钮、下拉框和弹窗
- `styles.css`
  负责全局主题、布局、模块样式、响应式适配
- `app.js`
  负责状态管理、模拟数据生成、路由切换、页面渲染、事件绑定、Canvas 图表绘制
- `vercel.json`
  用于部署到 Vercel 时把所有请求重写到 `index.html`

整体是典型的“单 HTML + 单 JS + 单 CSS”原型架构，部署和迁移成本很低。

## 4. 页面与模块组成

系统通过左侧导航切换不同视图，共包含 6 个业务页面：

### 4.1 实验室总览

首页聚合展示实验室整体运行情况，主要包括：

- 今日样本总数
- 在测样本数
- 已发布样本数量
- 异常样本数
- 异常分布 Top5
- SLA 告警
- 今日概览
- 最近 60 分钟样本动态折线图
- 各流程环节滞留情况

这一页的数据会按定时器持续刷新，是最“实时看板化”的部分。

### 4.2 测序下机数据质控

该模块聚焦下机批次和原始质控结果，页面功能包括：

- 批次切换
- 下机原始数据监控
- Index 组合产出热图
- 阳控/阴控在控情况
- 按项目查看整批次样本质控指标分布
- 单样本质控明细
- 箱型图放大和离群点点击联动

这是一个较完整的“批次级质控分析页”。

### 4.3 检测结果质量控制

该模块聚焦检测结果层面的统计分析，主要包括：

- 癌种筛选
- 变异类型筛选
- 热点位点筛选
- 批次热点变异检出率表格
- 特殊指标统计（MSI/TMB/HRD/LOH）
- 位点分级统计
- 阳性率趋势图
- 特殊指标分值趋势图
- 项目维度位点数统计
- VAF 散点图
- 按月位点数分布
- 按月 VAF 分布

这个页面是全项目里统计维度最多、图表最丰富的模块之一。

### 4.4 人效分析

该模块围绕人员与流程效率做分析，支持三种口径：

- 按批次
- 按月份
- 数据统计

主要内容包括：

- TAT 统计
- 人员吞吐量统计
- 项目 × 人员平均处理时长矩阵
- 人工删除位点数统计
- 异常样本数统计
- 月度箱型图趋势分析

这个模块既有表格，也有按月统计图，适合做团队效率和流程瓶颈分析。

### 4.5 科研画图服务

这是一个偏“服务工具型”的页面，主要流程是：

- 选择图表模板
- 下载数据模板
- 上传数据文件
- 生成图像
- 预览与下载

内置图表模板包括火山图、热图、生存曲线、散点图、柱状图、箱线图、折线图、韦恩图、PCA 图、相关性热图、森林图、气泡图等。

需要注意的是：当前版本虽然提供了文件上传入口，但生成图片时并不会真正解析上传文件，而是按所选模板绘制模拟示例数据。

### 4.6 实验室管理 alpha 版

该模块模拟了样本流转管理页面，功能包括：

- 样本流转概览
- 未发布样本状态查询
- 按条码号、检测编号、批次号、截止日期筛选
- 分页查看长表格
- 异常样本明细和预警等级展示

这个模块偏运营和管理端风格，更接近实际业务系统页面。

## 5. 核心架构说明

### 5.1 路由方式

项目使用 `hash` 路由切换页面，例如：

- `#/home`
- `#/module/qc-seq`
- `#/module/qc-result`

`app.js` 中的 `applyRoute()` 会：

- 解析当前地址 hash
- 高亮左侧导航
- 切换对应的 `.view`
- 更新页面标题和副标题

### 5.2 状态管理方式

项目没有引入外部状态管理库，而是在 `app.js` 中维护一个统一的 `state` 对象。主要分支包括：

- `state.kpi`
- `state.flow`
- `state.qcSeq`
- `state.qcResult`
- `state.productivity`
- `state.plotService`
- `state.labAlpha`

这种写法对于原型项目足够直接，优点是集中、易查；缺点是后期继续扩展时，单文件会越来越重。

### 5.3 渲染方式

每个模块都采用“状态 + 渲染函数”的模式，例如：

- `renderKpis()`
- `renderPipeline()`
- `renderQcSeq()`
- `renderQcResult()`
- `renderProductivity()`
- `renderPlotService()`
- `renderLabAlpha()`

`renderAll()` 会统一触发各模块刷新。页面上的大多数表格、卡片、筛选项，都是由 JavaScript 通过字符串模板动态写入 DOM。

### 5.4 事件绑定方式

项目主要使用两类事件绑定：

- 顶层按钮、分页按钮、上传控件等直接绑定
- 多数组件通过事件委托绑定到 `document`

例如：

- 批次切换
- 复选框/单选框筛选
- 下拉框切换
- 弹窗关闭
- 图表模板选择

这种方式对原生 JS 项目比较高效，也减少了重复注册事件。

## 6. 数据生成与模拟逻辑

本项目的数据几乎全部为模拟生成，核心特点如下：

### 6.1 使用伪随机种子

代码中使用了 `seededRand()` 和 `normalRand()` 来生成稳定但看起来真实的随机数据。这样可以让原型既有波动感，又不会完全失控。

### 6.2 初始化函数

系统启动时会执行一组初始化函数：

- `initSim()`
- `initQcSeqSim()`
- `initQcResultSim()`
- `initProductivitySim()`
- `initLabAlphaSim()`

它们分别负责生成首页、批次质控、结果质控、人效、实验室管理等模块的初始数据。

### 6.3 定时刷新

系统通过两个定时器工作：

- 每 1.5 秒执行一次 `tickSim()`，推动首页实时数据变化并整体重绘
- 每 1 秒执行一次 `renderTime()`，更新右下角时间显示

需要注意的是，`tickSim()` 主要更新首页实时相关数据，不会让所有业务模块都发生真实联动。

## 7. 图表实现说明

项目图表全部使用原生 `canvas` 绘制，没有依赖 ECharts、Chart.js 等第三方库。

当前已实现的图表类型包括：

- 折线图
- 面积图
- 箱型图
- 带离群点箱型图
- 散点图
- 热图
- 柱状图
- 双箱型图
- VAF 分布图
- 森林图
- 生存曲线
- 火山图
- 韦恩图
- 相关性热图
- 气泡图

优点：

- 无额外依赖
- 可完全按业务需要定制
- 适合原型和固定场景

缺点：

- 代码量较大
- 图表逻辑都堆积在 `app.js` 中
- 后期维护、复用和测试成本会逐步上升

## 8. 样式与界面特征

页面整体采用深色大屏/看板风格，样式特点包括：

- 深色背景 + 渐变光晕
- 卡片化布局
- 左侧固定导航
- 顶部吸附式操作栏
- 多种胶囊标签、状态徽章、风险色
- 统一的圆角和玻璃拟态风格

响应式方面，`styles.css` 中提供了两个主要断点：

- `max-width: 1180px`
- `max-width: 880px`

在窄屏下会发生：

- 首页卡片改为两列或单列
- 侧边栏转为顶部横向导航
- 筛选区纵向排列
- 图表和表格区域适配小屏

因此，这个原型不是只面向大屏，也考虑了常规桌面和移动窄屏的可访问性。

## 10. 当前实现的亮点

从原型质量来看，这份代码已经具备以下优点：

- 模块覆盖面广，几乎把实验室看板的核心场景都串起来了
- 数据模拟相对真实，不是简单写死数字
- 图表类型丰富，适合演示统计分析场景
- 交互较完整，包含筛选、切换、放大、下载、分页等功能
- 页面视觉统一，具备较强的汇报展示感

## 11. 当前实现的限制

这部分对后续产品化很重要：

- 没有后端接口，所有数据都是前端模拟
- 没有用户体系、权限控制、登录流程
- 没有真正的数据持久化
- 科研画图上传文件不会被解析，当前只是“上传占位 + 模拟出图”
- 所有逻辑集中在一个 `app.js` 文件中，文件体量已经很大
- 页面渲染大量依赖字符串拼接，后续维护容易变复杂
- 缺少自动化测试和模块边界约束
- `重置模拟` 当前会重建首页、质控和人效数据，但不会重新初始化 `labAlpha` 数据

## 12. 后续优化建议

如果后面要继续往正式系统推进，建议分阶段处理：

### 第一阶段：代码拆分

- 将 `app.js` 拆成按模块组织的多个文件
- 把通用工具函数、绘图函数、数据模拟函数拆开
- 将状态定义单独维护

### 第二阶段：真实数据接入

- 为各模块定义统一的数据接口结构
- 将模拟数据层替换为 API 数据层
- 保留模拟数据开关，方便演示和测试

### 第三阶段：组件化改造

- 如果功能继续扩展，建议迁移到 Vue 或 React
- 将表格、卡片、筛选器、图表容器做成组件
- 将模块页面独立成单独文件，降低维护成本

### 第四阶段：业务能力增强

- 为科研画图模块增加真实文件解析能力
- 为实验室管理模块增加排序、导出、详情页
- 为质控模块增加异常追踪闭环和处理记录
- 为人效模块增加时间筛选、组织维度、人员维度联动

## 13. 结论

这套代码已经不是简单的静态页面，而是一份完成度较高的业务原型。它的价值主要体现在两个层面：

- 对外，可以用于方案汇报、培训演示、需求确认
- 对内，可以作为后续正式产品开发的业务蓝本

如果仅作为原型展示，这套实现已经足够完整；如果要走向生产系统，下一步最关键的是“模块拆分 + 真实数据接入 + 组件化重构”。
